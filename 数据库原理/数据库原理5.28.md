## 一、数据定义

### 1. 概述与SQL语言分类

数据库表格还有一个索引等等一些定义，还有一些更改的应用操作，我也会说到。另外的话，可能会涉及到这一章比较关键的就是一个视图这个概念，然后后面的话就是游标、存储过程还有这个触发器，这三个针对性的模块。

我们这边数据定义的话就稍微过一下，主要涉及到一些数据库、基本表、视图和索引的创建、修改和删除。这张表可以统计出来：创建的话主要用的是`CREATE`这个关键词，对于数据库就是`DATABASE`，然后`TABLE`，然后`VIEW`和`INDEX`分别对应基本表、视图和索引。修改的话主要是`ALTER`，删除的话主要是`DROP`关键词。

### 2. 数据库的创建、修改与删除

数据库定义的话，整体的规则主要是涉及到一个数据文件和日志文件这两类。定义就是我们整个`CREATE DATABASE`，就是创建数据库的时候，后面我们结合相对应的语句其实就可以看得到。总体就是`CREATE DATABASE`，然后后面是数据库的名字，方括号里面就是你可以自己定义的，这个数据库管理系统会处理技术问题。

创建数据库以后会涉及到两个，刚才说的一个是存储文件，就是我们这里的存储文件的具体位置，`specific`就是具体的路径名。然后这个`LOG`就是我们的整个日志文件怎么存储，主要是一些规则，没有什么特别的。

修改数据库的话，就是我们可能需要做一些增加文件、增加一些数据表，然后把这个文件从原始的某个位置移到位置B，可能还会删除到一些文件或者修改一些文件。主要就是用到这个`ALTER`，整体架构都一样。

这边就是一个例子，比如我们修改这个`my_temp_db`，这个是我们自己创建的数据库名字，然后做了一个数据库的修改，具体的就是把`temp_history`这个文件，把它的尺寸数值改了。删除数据库的话就更简单了，就是`DROP DATABASE`就可以了。

### 3. 基本表的创建、修改与删除

基本表的话也是类似的，只不过把前面的`DATABASE`关键词改成`TABLE`，就是`CREATE TABLE`，然后`ALTER TABLE`和`DROP TABLE`。

**创建表：** 我们每张表里面有这个列，说白了就是学号、姓名等等这些信息，可能会需要涉及到学号是什么类型，然后姓名可能是字符型的，出生日期可能是日期型的，这些都是在创建数据表里面来定义的。整体创建数据表就是`CREATE TABLE`，然后这个是我们的表名，比如说`student`这个表，然后可能会涉及到学号，它的一个`data type`是`INT`（比如说假设），然后`student_name`比如说是个`CHARACTER`多少位。这里就是可能有各种各样的限制，比如说学号不能重复，我们就给它一个`PRIMARY KEY`的限制；另外的话就是可能有一些外码的限制，外码的话又会涉及到参照表，就会有这个`REFERENCE`。

这边就是一个例子，就是我们建立课程表`course`，然后这个学分等等一些信息，这边是它的数据类型，不能重复各种各样的限制，然后最后的主码在这里是课程号，然后外码的话会涉及到一个先修课程，它的参照表是我们课程表本身。

**修改表：** 修改表的话其实也比较容易，可能会涉及到一个增加列、增加约束或者删除约束等等。这边有些相关的例子，我们就不展开了。

**删除基本表：** 删除基本表的话就是直接用`DROP TABLE`。这边会有一个`RESTRICT`跟`CASCADE`，这个是有些数据库系统会有的功能，但是一般不大常用，尤其像MySQL数据库它都没有定义这个东西。但这个`RESTRICT`跟`CASCADE`，尤其是后面在权限管理、安全性管理的时候可能会涉及到，到时候我们在第九章的时候会再详细介绍一下。包括这里视图也会涉及到，就是删除或者撤销某个权限以后，可能会涉及到进一步的一些数据表格或者权限的修改。但在删除表这个场景下，一般不会过多用到，甚至SQL Server都不支持这个选项，所以虽然语法上有，但实际不常用。

### 4. 索引

接下来就是索引。我们的数据怎么存放，每条记录怎么存放，然后在查询的时候怎么有一些有效的排序方式。存放和查询，这两个就引出索引的两类：一个叫做聚集索引，一个叫做非聚集索引。

一般聚集索引只有一个，因为它跟数据怎么存放是相关的。比如说我们按学号做索引，如果把学号定义为聚集索引的话，那整个学生表就按学号的顺序进行排序。非聚集索引，比如说我们可以按学生姓名给他做一个非聚集索引，这样的话到后续做查询的时候会比较方便，根据每个学生姓名就可以查询每一条记录。

总体索引的一个样式，左边这个表格相当于是一个索引的形式，这里我们可以看到它其实是非聚集索引，因为整个这个索引并没有决定表格的数据物理存放顺序。索引一般是按照B+树的结构进行组织的，B+树和哈希索引是我们最常见的索引组织方式。

建立索引的规则或者语法：`CREATE`，然后指定`CLUSTERED`或`NONCLUSTERED`（聚集索引或非聚集索引），然后它的对应列属性是哪一个，然后`ASC`就是升序，`DESC`就是降序。最后可能会涉及到一个`FILEGROUP NAME`，就相当于我们索引本身存在哪个文件组里。基本上有个例子，就是建立一个非聚集索引，按班级的属性。另外就是非聚集索引，你可以建多个，但太多的话，系统存储开销会越来越高。

## 二、数据更新

### 1. DML概述与SQL语言区分

接下来的话就是会涉及到数据的更新，就是数据的增删改。因为我们SQL第三章就最后开始是查询，这个也会涉及到增删改，当时提到过一些，在第七章就是整个课程的中后段会介绍具体系统性的增删改。

数据更新对应的是DML，就是数据的增删改查四类操作中的增删改。但查询是检索，是我们最主要的内容，所有的增删改加起来可能还不如检索内容的一半。检索或查询是我们第三章的内容，一定要再强调一下，整个当时第三章的一大章节都是我们检索或者查询的内容。现在这个是7.2小节，就是我们更新的系统介绍。

那么数据更新——增加、删除、修改——对应的是`INSERT`、`DELETE`跟`UPDATE`。这个地方是要区分一下，因为我们在考试里面会涉及到在第三大题后面里面会有一些建立一个表格或者是修改一些数据，要用`INSERT`，不要跟前面的`CREATE`、`ALTER`跟`DROP`搞混了，这个是另外一组关键词。

表格和数据库的创建是`CREATE`，修改是`ALTER`，不是`UPDATE`；数据更新才用`UPDATE`。这个地方一定要区分清楚，你不要到时候说修改一个表格，写了一个`UPDATE TABLE`，那是不对的，应该用`ALTER`。

### 2. 插入数据（INSERT）

插入数据首先这种插入数据就是主要可以插入单个元组，也可以插入一个子查询。

**插入单条记录：** 固定搭配就是`INSERT INTO ... VALUES ...`。`VALUES`后面跟的就是具体的插入的元素信息。比如说我们插入学生表里面插入一个新学生信息，就是类似的，下面考试的题目就是"将一个新学生元组插入到学生表中"。主要是这个固定搭配。

**插入多条记录（子查询）：** 如果是插入多条数据，可以在`INSERT INTO`后面直接替换成一个子查询的形式，`SELECT ... FROM ... WHERE ...`，就是一个完整的查询语句。比如说把这个少数民族同学的选课信息插入进去，用的就是子查询的方式。

### 3. 删除数据（DELETE）

删除数据就是`DELETE FROM`。删除学号为160001同学的信息：`DELETE FROM 表格 WHERE 学号 = '160001'`。如果是删整张表的所有记录，就是`DELETE FROM`不加`WHERE`；如果是删除具体的一条表记录，就要加`WHERE`条件。

### 4. 修改数据（UPDATE）

修改就是`UPDATE`。相当于我们更新一个表格，然后`SET`后面就是我们更新的相应信息。假设我们想把张三改成张四，就是`UPDATE 表格 SET 姓名 = '张四'`。

后面会有例子，就是加入"将红敏同学的这个学期选课的成绩改为88"。就是在`SCORE`表格里，`SET SCORE = 88`。每一条就是现在这`SCORE`列中的某一条信息，你要指明是哪一条数据，通过`WHERE`来进行限定。

这里还需要注意一下：`UPDATE`跟`DELETE`和`INSERT`有一个区别。`INSERT INTO`后面跟的子查询是`SELECT FROM WHERE`一个完整的形式，但是`UPDATE`跟`DELETE`的话，一般只是做一个条件性的判断，`UPDATE`直接跟`SET`，然后`FROM`是可选的，不一定要显示地把所有表做连接操作。因为这里我们可以看到，学号跟学期号都属于`SCORE`，但是学生姓名其实属于学生表。在语义表示已经很清晰的情况下，连接操作可以不是显式的。

**数据更新小结：** 整个数据更新——插入、删除、修改，对应的是`INSERT`、`DELETE`跟`UPDATE`，这个一定要跟前面的`CREATE`、`ALTER`、`DROP`区分开来。插入一条记录就是`INSERT INTO ... VALUES ...`；多条记录就是在`INSERT INTO`后面直接跟子查询；删除表记录就是`DELETE FROM`，加`WHERE`条件；修改就是`UPDATE ... SET ...`。

## 三、视图

### 1. 视图的概念与作用

视图的话，这里也是个系统性的介绍。首先视图它是一个虚表，它对应的是我们的外模式。数据库系统三级模式：内模式、模式、外模式。视图就是外模式，是基于应用的需要来定义的。

视图的主要作用之前介绍过，就是体现逻辑独立性，还有安全性的保护。当然对于视图本身，它是一个虚表，所以一般不太建议做更新操作，更新都是一个受限的更新，后面我会再提到。

### 2. 视图的创建

创建视图的话，基本表是用`CREATE TABLE`，这里就是`CREATE VIEW`。因为视图是从基本表中选择一些数据集或者一些特定条件记录，这里会涉及到`CREATE VIEW ... AS`这个关键词。

比如创建一个"二三级同学"的视图：`CREATE VIEW 视图名 AS SELECT * FROM student WHERE ...`。

另外需要注意的就是`WITH CHECK OPTION`。因为我们创建了视图以后还会做一些增删改操作在上面，但是视图本身是一个虚表，它是建立在基本表的基础上的。如果我们对于视图做增删改，如果不符合这个视图的查询条件的时候，加了`WITH CHECK OPTION`就会拒绝做增删改。如果没有加，是可以操作的，但在查询的时候不会显示出来。

**举例说明：** 创建一个视图，查询条件是`birthday`是1999年。如果没加`WITH CHECK OPTION`，我们可以插入一个98年的同学信息，这是可以插入进去的。但在做视图查询的时候，这条信息不会显示出来。如果加了`WITH CHECK OPTION`，这条信息就插不进去了，因为它98年的不符合查询条件，直接拒绝操作。

### 3. 视图的查询

查询肯定也是视图最主要的功能。整个查询跟基本表没有特别大的区别，因为视图本身也是一张表，就是正常的`SELECT`，各种各样的连接或者自然连接都可以实现，第三章的内容可以直接套用在视图上。只不过就是`FROM`里面的这个表格，它是一张虚表，是一个视图，不是一个基本表。

但是对我们数据库系统来说会涉及到额外的一个变化，因为视图本身是虚表。我们对数据进行查询以后，整个数据库系统会做三个步骤：第一步先做有效性检查，查一下这个视图存不存在。如果存在的情况下，就取出创建视图的语句，然后把创建视图的语句和用户的查询语句做一个合并和融合，来生成真正的针对基本表的查询。

比如说，视图是出生年是99年的，然后用户的查询是`class number = 'xxx'`。对于这个例子，就是相当于把创建视图的语句跟上面的用户查询去做一个合并融合，生成针对基本表的查询。

### 4. 视图的更新

增删改数据就是视图更新，这里要特别着重强调：**尽量不要对视图进行更新**。因为视图本身是一个虚表，你要做更新，就去基本表更新。但是在语法上，或者操作上还是允许的，前提是这个视图必须是**行列子集视图**。

**行列子集视图的定义：** 基于一个基本表，没有任何聚合函数（如`SUM`、`COUNT`、`AVG`、`MAX`），并且保留了主码属性。这类行列子集视图是可以做更新的。因为我们更新这类视图的某一条信息，只会影响到建立这个视图的单张基本表，不会影响到其他的表。

其他所有涉及到了统计信息、聚合函数的信息，或者涉及到多表连接的视图，一般是不允许更新的。总的来说，一般不建议对视图做更新操作，视图主要是在查询用的。

### 5. 视图的删除与级联影响

删除视图的话就是`DROP VIEW`。这里稍微注意一下：如果你删除了某个基本表，那么这个基本表导出的所有视图都要删除，但是删除必须要显式地用`DROP`语句进行。另外，如果视图B建立在视图A的基础上，那么删除掉视图A的话，视图B也会自动删除掉。这个地方要稍微注意一下。

## 四、T-SQL语言扩展

### 1. T-SQL概述

接下来就是我们的下半部分，我称为T-SQL。T-SQL说白了就是MySQL或者SQL Server这些系统，针对于传统或经典的SQL语言做的一些扩充性的设计，可能会有一些特殊的函数，比如说之前大家还有印象的`YEAR()`，它是取出生日对应的年份。还有一些就是我们可以加一些传统程序语言的东西，比如`IF ELSE`，甚至`FOR`循环等等。

由T-SQL语言引出来三个模块：游标、存储过程和触发器。这三个就可以实现一些稍微高级一点的功能。

### 2. T-SQL的扩充功能

T-SQL全称是Transact-SQL，它其实是基于SQL本身做了延伸性的设计。可能会涉及到变量的定义——局部变量、全局变量，还有数值列表、赋值等等。所以如果有额外的局部变量，主要是为了跟后面的`IF ELSE`，或者是存储过程和游标配合。

函数本身，T-SQL对传统SQL增加了比如说数学函数（四舍五入、向上取整、向下取整等）、字符串函数、日期和时间函数等，这些都是SQL里面没有的，但在T-SQL里面是可以用的。还有一些其他的系统函数，比如`CONVERT`做强制类型转换等。

另外还有一些流程控制，就是`IF ELSE`、`CASE`、`CONTINUE`、`BREAK`、`WHILE`等，这些都是传统程序语言里面的，T-SQL已经有一部分应用化了。具体程序示例就是查询成绩表中的最高成绩，如果大于95输出'Very Good'，大于80输出'Good'，这里面就加了局部变量和流程控制语句。

## 五、游标

### 1. 游标的概念

游标的话，它可以认为是一个缓存。数据缓冲区中，把一些查询结果存在里面，然后去做类似于迭代器的功能。就是相当于把一组数据，比如说99年出生的所有学生的数据，我们可以存到一个游标里面。然后这个游标自动地进行迭代，一条一条信息往下走。

比如说三条数据就可以存到游标里面，把它放在一个数据缓冲区里。游标相当于一个迭代器的作用，它可以自动地一条一条信息往下做一个迭代更新。

### 2. 游标的五个步骤

整个游标会涉及到五个步骤：定义游标、打开游标、逐行提取游标中的记录、关闭游标、释放游标程序空间。

**（1）定义游标：** 固定规则。`DECLARE 游标名 CURSOR FOR`，`CURSOR`是必须有的固定关键词，游标名可以自己定义，`FOR`后面是正常的SQL语句，因为我们整个游标是存储查询出来的数据集的。

**（2）打开游标：** 打开以后它就默认指向游标器中的第一个元素。这点要强调一下：你一旦打开，它就是指向第一个元素，而不是说打开以后还需要你去把它定义到或者指向第一个。这个顺序问题要注意。

**（3）逐行提取：** 正常的获取当前的元组，`FETCH 游标名 INTO 变量名`。每执行一次`FETCH`，把这一条数据存到对应的局部变量里面，然后游标会自动移到下一条记录。它不需要任何`i = i+1`的操作，是自动化的，每再执行一次，它就自动往下走。

当前的游标指向元组的每个属性必须分别用一个变量接收。因为可能会有多个属性，比如说学号和姓名分别存到两个变量里面。

当游标已经遍历完所有记录，不可再次读取，必须要关闭游标，然后重新再打开。它只能往下，不能往上后退。可以通过检查`@@FETCH_STATUS`全局变量来判断是否读取完毕，等于0就是成功，-1就是失败，-2就是此行不再有效。可以基于这个全局变量做判断，如果出错的话可以加一些补偿或备选措施。

**（4）关闭与释放：** 正常的关闭游标就是`CLOSE`，关闭以后还要`DEALLOCATE`释放程序空间。

### 3. 游标举例

结合一个例子：创建一个游标，逐行显示选出了计算机原理课程的选择情况。

首先定义游标，`DECLARE 游标名 CURSOR FOR`，查询语句是从三张表中查询学生姓名、成绩、学期、课程名等，条件是"计算机原理"课程，并且按一定顺序排列。

定义好游标以后，定义两个局部变量来统计成绩。我们可以利用游标来定义`cnt_score`和`sum_score`，一个是课程计数，一个是总分，后续用来求平均分。两个是全局变量，后续的平均分计算通过游标来进行迭代处理。

具体程序流程是这样：
- 先声明相对应的局部变量：学生姓名、成绩和对应的学号，这三个是从游标中直接提取出来的信息。
- 然后打开游标，打开以后自动指向第一行。
- 先执行一次`FETCH`（处理第一行信息），把三个变量填好。
- 然后进入`WHILE`循环（条件是`@@FETCH_STATUS = 0`），每循环一次就处理一行，把分数加到`sum_score`里，计数器`cnt_score`加一，然后再次`FETCH`自动进入下一行。
- 当所有记录处理完毕后，计算平均分（总分除以选课人数），输出结果。
- 最后关闭游标，释放游标空间。

**关于为什么先执行一次`FETCH`再进入循环：** 因为`OPEN`游标以后自动指向第一行，所以先执行一次`FETCH`处理第一行信息，然后进入`WHILE`循环处理剩下的所有记录。这其实就是执行顺序的问题，因为游标默认打开就指向第一个元素。

另外，对于游标，我们还可以做删除或修改操作。对当前游标指向的当前元组，可以做删除或修改。

## 六、存储过程

存储过程就是更加通用一点，有点类似于我们自定义的函数。我们先定义一个存储过程，在使用的时候就调用一下就可以了，对应函数的一个作用。

存储过程的作用：可能有安全性保护，然后适合统计和查询，还能节省重复编译量。假如某一条查询语句经常使用，我们就可以把这个`SELECT FROM WHERE`封装成一个存储过程，做成一个函数，那么每次使用的时候，我们就是通过参数调用的形式，节省了每次传输完整查询语句的开销。

创建存储过程的语法：`CREATE PROCEDURE 过程名 @参数 参数类型 AS ...`。`PROCEDURE`就是存储过程的关键词，后面括号里的参数可以任意定义，然后`AS`后面就是SQL语句。

**例子：** 输入某个同学的学号，查询该同学的平均分。这个查询语句如果是会多次使用的，一般会把它做成存储过程。把学号定义为参数，后续调用的时候，传入学号就能查询对应同学的信息。`CREATE PROCEDURE 过程名 @学号 INT AS SELECT ... FROM ... WHERE 学号 = @学号`。

唯一需要注意的是，常规的SQL里面我们查询某同学就是`WHERE 学号 = 1`，但是封装成存储过程以后，就是把常量替换成了一个参数变量，其他没有任何区别。无非就是把一个比较常用的SQL查询语句封装成一个存储过程，可以理解成函数的形式。

调用的时候，用`EXECUTE`执行，传入实际的参数，就非常像正常程序语言里面的函数定义加上函数调用。

存储过程里面还可以进一步嵌套一个游标，这个我们时间关系就不展开了。

---

## 七、触发器

触发器本身是一个事件驱动的，它是由服务器自动激活的特殊存储过程，可以认为是特殊的一个函数。它有一些特定条件才会激活。

触发器跟后面我们的事务定义（提交、回滚操作）是基本相关的。最常用的是通过触发器来实现数据的约束。比如最开始我们插入98年的同学信息到99年的视图里面，正常是不能够插入的。一种方式是用视图的`WITH CHECK OPTION`来实现，另外一种方式就是通过触发器。一旦插入了这条信息，它不满足给定的条件，就可以通过触发器来实现一个回滚操作。所以触发器本身是一个特殊的存储过程，它跟后面我们介绍的事务是基本相关的。

触发器的实现一般会涉及到两张系统表：`deleted`表和`inserted`表。这两张表跟我们的插入、删除和修改是息息相关的。

- 当我们**插入**一条数据时，系统会自动在`inserted`表里增加一条记录。
- 当我们**删除**一条数据时，系统会自动在`deleted`表里增加一条记录。
- 当我们**修改**数据时，比如说把张三改成李四，就可以理解为把张三这条信息删除掉，然后把李四这条信息插入进来。所以`deleted`表里会加张三，`inserted`表里会加李四。

这两个表是系统维护的，不是用户自己定义的。我们整个触发器都是基于这两张表来进行相应的实现。SQL Server、Oracle和其他数据库都有对应的这种机制。
